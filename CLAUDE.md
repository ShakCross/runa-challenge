# CLAUDE.md вЂ” HR Assistant Agent Project Context

## Project Overview

HR Assistant Agent for the Time Off Module. Full-stack app with an LLM-powered conversational agent that helps employees check vacation days, request time off, ask about company policies, and check team calendars.

## Tech Stack

- **Backend**: Node.js + Express + TypeScript
- **Frontend**: React + TypeScript + Vite
- **LLM**: OpenAI GPT-4o-mini (via openai npm package)
  - Chosen for: excellent tool/function calling, cost-effective, fast, strong multilingual support
- **State**: In-memory session store (no database needed for this scope)

## Tools (4 total)

### 1. getRemainingVacationDays(employeeId: string)
- Returns mock vacation balance: { total, used, remaining, pending }
- Simple lookup from in-memory mock data

### 2. requestTimeOff(employeeId: string, startDate: string, endDate: string)
- Validates: dates in the future, startDate <= endDate, sufficient balance
- Deducts business days from balance
- Returns confirmation or error message
- Date format: YYYY-MM-DD

### 3. getCompanyPolicy(topic: string)
- Supported topics: "vacation", "sick_leave", "remote_work", "holidays", "parental_leave"
- Returns mock policy text for each topic
- If unknown topic, returns list of available topics

### 4. getTeamCalendar(employeeId: string, month?: string)
- **Why**: Before requesting time off, employees need to check team coverage to avoid conflicts
- Returns list of upcoming team members' approved time off
- Helps plan time off without leaving the team understaffed
- Natural complement to the time-off request flow

## Architecture Decisions

### Agent Loop
1. User sends message в†’ backend receives via POST /api/chat
2. Session history is loaded (or created) from in-memory store
3. Message is appended to conversation history
4. LLM is called with: system prompt + conversation history + tool definitions
5. If LLM returns tool_calls в†’ execute tools в†’ append results в†’ call LLM again
6. Loop continues until LLM returns a final text response (no more tool_calls)
7. Response + any tool call metadata returned to frontend

### Session Management
- Sessions identified by sessionId (UUID generated by frontend on first load)
- Conversations stored in a Map<sessionId, ChatMessage[]>
- No persistence needed (in-memory is fine for this scope)

### Employee Identity
- Assumption: employeeId is provided as part of the session/request context
- For demo: default employeeId = "EMP001" (configurable in frontend)
- Mock data includes 5 employees in the same team

### Mock Data
Five employees in "Engineering" team:
- EMP001: Alice Johnson вЂ” 20 total days, 5 used
- EMP002: Bob Smith вЂ” 15 total days, 10 used
- EMP003: Carol Williams вЂ” 20 total days, 2 used
- EMP004: Dave Brown вЂ” 18 total days, 18 used (no balance!)
- EMP005: Eve Davis вЂ” 22 total days, 8 used

### Ambiguity Handling
- System prompt instructs the agent to ask clarifying questions when input is vague
- Examples: "I want time off" (missing dates) в†’ agent asks for dates
- Unknown employee в†’ agent asks to verify ID
- Past dates в†’ agent informs and asks for valid dates

### Multilingual Support
- System prompt instructs agent to respond in the same language the user writes in
- GPT-4o-mini handles this natively

## API Contract

### POST /api/chat
Request:
```json
{
  "sessionId": "uuid-string",
  "employeeId": "EMP001",
  "message": "How many vacation days do I have left?"
}
```

Response:
```json
{
  "response": "You have 15 vacation days remaining out of 20 total.",
  "toolCalls": [
    {
      "tool": "getRemainingVacationDays",
      "args": { "employeeId": "EMP001" },
      "result": { "total": 20, "used": 5, "remaining": 15, "pending": 0 }
    }
  ]
}
```
